错误类型解析

error 接口

Go 继承了 C，以返回值为错误处理的主要方式。但与 C 不同的是，在 Go 的惯用法中，返回值不是整型等常用返回值类型（errno），而是用了一个 error（interface 类型）。

1
2
3
4
5

// The error built-in interface type is the conventional interface for
// representing an error condition, with the nil value representing no error.
type error interface {
	Error() string
}

它只有一个方法 Error，只要实现了这个方法，就是实现了error。现在我们自己定义一个错误试试。

 1

func New(text string) error {
	return &errorString{text}
}

type errorString struct {
	s string
}

func (e *errorString) Error() string {
	return e.s
}

变成以上这样，我们就可以通过New函数，辅助我们创建不同的错误了，这其实就是我们经常用到的errors.New函数.

在 Go 标准库中，Go 提供了两种创建一个实现了 error interface 的类型的变量实例的方法：errors.New 和 fmt.Errorf

1
2

errors.New("your first error code")
fmt.Errorf("error value is %d\n", errcode)

这两个方法实际上返回的是同一个实现了 error interface 的类型实例，这个 unexported 类型就是 errorString。顾名思义，这个 error type 仅提供了一个 string 的 context！

如果我们想对返回的error附加更多的信息后再返回，我们先通过Error方法，取出原来的错误信息，然后自己再拼接，再使用errors.New函数生成新错误返回。

Go中error类型的nil值和nil

先看C语言中的类似问题：空字符串。

1
2
3

const char* empty_str0 = "";
const char* empty_str1 = "\0empty";
const char* empty_str2 = NULL;

以上3个字符串并不相等，但是从某种角度看，它们都是对应空的字符串。

  • empty_str0 指向一个空的字符串，但是empty_str0本身的值是有效的。
  • empty_str1 指向一个非空的字符串，但是字符串的第一个字符是'\0'。
  • empty_str2 本身是一个空的指针。

Go的error是一个interface类型，error的nil问题和C语言的字符串类似。

参考官方的error文档说明：

http://golang.org/doc/go_faq.html#nil_error

在底层，interface作为两个成员实现：一个类型和一个值。该值被称为接口的动态值， 它是一个任意的具体值，而该接口的类型则为该值的类型。对于 int 值3,一个接口值示意性地包含(int, 3)。

只有在内部值和类型都未设置时(nil, nil)，一个接口的值才为 nil。特别是，一个 nil 接口将总是拥有一个 nil 类型。若我们在一个接口值中存储一个 *int 类型的指针，则内部类型将为 *int，无论该指针的值是什么：(*int, nil)。 因此，这样的接口值会是非 nil 的，即使在该指针的内部为 nil。

下面是一个错误的错误返回方式：

1
2
3
4
5
6
7

func returnsError() error {
	var p *MyError = nil
	if bad() {
		p = ErrBad
	}
	return p // Will always return a non-nil error.
}

这里 p 返回的是一个有效值（非nil），值为 nil。 类似上面的 empty_str0。

因此，下面判断错误的代码会有问题：

1
2
3
4
5

func main() {
	if err := returnsError(); err != nil {
		panic(nil)
	}
}

针对 returnsError 的问题，可以这样处理（不建议的方式）：

1
2
3
4
5

func main() {
	if err := returnsError(); err.(*MyError) != nil {
		panic(nil)
	}
}

在判断前先将err转型为*MyError，然后再判断err的值。 类似的C语言空字符串可以这样判断：

1
2
3

bool IsEmptyStr(const char* str) {
	return !(str && str[0] != '\0');
}

但是Go语言中标准的错误返回方式不是returnsError这样。 下面是改进的returnsError：

1
2
3
4
5
6

func returnsError() error {
	if bad() {
		return (*MyError)(err)
	}
	return nil
}

错误处理策略

go语言中一般有三种错误处理策略：

  1. 返回和检查错误值：通过特定值表示成功和不同的错误，上层代码检查错误的值，来判断被调用func的执行状态
  2. 自定义错误类型：通过自定义的错误类型来表示特定的错误，上层代码通过类型断言判断错误的类型
  3. 隐藏内部细节的错误处理：假设上层代码不知道被调用函数返回的错误任何细节，直接再向上返回错误

返回和检查错误值

这个名字来源于计算机编程中使用特定值来表示无法进一步处理的做法。因此，对于Go我们使用特定的值来表示错误。例子包括类似于像包io.EOF或常量那样的低级错误syscall，比如syscall.ENOENT。甚至有用sentinel error，表示没有发生错误，例如go/build.NoGoError, and path/filepath.SkipDir 。

使用sentinel错误处理是最不灵活的错误处理策略，因为调用者必须使用相等运算符将结果与预先声明的值进行比较。当你想提供更多的上下文时，这会产生问题，因为返回不同的错误将会破坏相等性检查。即使是fmt.Errorf用来向错误添加一些上下文的含义也会破坏调用者的相等性测试。相反，调用者将不得不看的输出error的Error方法，看它是否一个特定的字符串相匹配。

这种策略是最不灵活的错误处理策略，上层代码需要判断返回错误值是否等于特定值。如果想修改返回的错误值，则会破坏上层调用代码的逻辑。

1

buf := make([]byte, 100)
n, err := r.Read(buf)   // 如果修改 r.Read，在读到文件结尾时，返回另外一个 error，比如 io.END，而不是 io.EOF，则所有调用 r.Read 的代码都必须修改
buf = buf[:n]
if err == io.EOF {
    log.Fatal("read failed:", err)
}

另外一种场景也属于这类情况，上层代码通过检查错误的Error方法的返回值是否包含特定字符串，来判定如何进行错误处理。

顺便说一句，不应该检查error.Error方法的输出。该Error方法仅仅提供可视化，而不是代码逻辑判断。该字符串的内容属于日志文件，或显示在屏幕上。有时这是不可能的，正如有人在twitter上指出的那样，这个建议不适用于编写测试。比较字符串形式的错误是一种坏的代码风格，你应该尽量避免它。

如果你的公共函数或方法返回特定值的错误，那么该值必须是公开的，并且应该有文档说明来增加API完整性。如果你的API定义了一个返回特定错误的接口，则该接口的所有实现都将被限制为仅返回该错误类型，即使它们可能提供更多描述性错误。我们来看io.Copy函数，需要一个实现io.Reader接口的对象，对象返回io.EOF的信号，是通知调用者没有更多的数据，但这不是一个错误。

Sentinel errors create a dependency between two packages

到目前为止，Sentinel错误值最糟糕的问题是它们在两个包之间创建源代码依赖关系。作为一个例子，为了检查错误是否等于io.EOF，你的代码必须导入这个io包。这个具体的例子听起来不是很糟糕，由于它很常见，但想象一下,当项目中的许多软件包导出错误值时，导致耦合度上升，项目中的其他软件包也必须导入依赖包以检查特定错误情况。在参与一个大型项目中，不良设计的幽灵（以导入循环的形式）是我们谨慎的事情。

结论：避免Sentinel错误

所以，建议是避免在你编写的代码中使用标记错误值。在标准库中，有少许的示例使用，但这不应该值得我们借鉴。如果有人要求你从你的包中导出一个错误值，你应该礼貌地拒绝，而是建议一种替代方法，比如将在下面讨论的方法。

自定义错误类型

如果要提供额外的 error context，我们可以定义自己的实现 error 接口的类型；如果这些类型还是 exported 的，我们就可以用 type assertion or type switch 来判断返回的错误码类型并予以对应处理。

这种方式相比于“返回和检查错误值”，很大一个优点在于可以将底层错误包起来一起返回给上层，这样可以提供更多的上下文信息。

Problems with error types

由于调用者可以使用类型断言或类型切换，错误类型必须公开。如果你的代码实现了其需要特定错误类型的接口，则该接口的所有实现者都需要依赖于定义错误类型的包。这种对包类型的深入了解可以与调用者形成强大的耦合，从而形成一个脆弱的API。

结论：避免error types

虽然错误类型比标记错误值更好，但由于它们可以捕获关于错误的更多上下文，错误类型与标记错误值存在共同的问题。所以我的建议是再次避免错误类型，或者至少避免让它们成为公共API的一部分。

隐藏内部细节的错误处理

这种策略之所以叫“隐藏内部细节的错误处理”，是因为当上层代码碰到错误发生的时候，不知道错误的内部细节。作为上层代码，你需要知道的就是被调用函数是否正常工作。 如果你接受这个原则，将极大降低模块之间的耦合性。

现在我们来看第三类错误处理。在我看来，这是最灵活的错误处理策略，因为它要求代码和调用者之间的耦合最少。我将这种风格称为不透明错误处理，因为当你知道发生错误时，你无法看到错误内部。作为调用者，你所知道的操作结果只是它的工作原理，或者没有。这就是所有不透明的错误处理 - 只是返回错误而不假设其内容。如果你采用这种方式，那么错误处理作为调试辅助工具变得更加有用。

1

import “github.com/quux/bar”

func fn() error {
    x, err := bar.Foo()
    if err != nil {
            return err
    }
    // use x
}

例如，Foo函数不能保证它会在错误的情况下返回什么。现在，Foo可以自由注释通过附加上下文传递的错误，而不会违反与调用方（fn）的关系。

Assert errors for behaviour, not type

在少数情况下，这种错误处理的方法是不充分的。例如，与流程之外的世界进行交互（例如网络活动）要求调用者调查错误的性质，以决定重试操作是否合理。在这种情况下，我们可以断言错误实现了特定的行为，而不是断言错误是特定的类型或值。考虑这个例子：

type temporary interface {
        Temporary() bool
}

// IsTemporary returns true if err is temporary.
func IsTemporary(err error) bool {
        te, ok := err.(temporary)
        return ok && te.Temporary()
}

我们可以传递任何错误给IsTemporary函数来确定错误是否可以重试。如果错误没有实现temporary接口，也就是说，它没有Temporary方法，那么错误不是暂时的。如果错误确实实现了Temporary，调用者能调用重试操作。这里的关键是这个逻辑可以在不导入定义错误的包或实际知道任何关于err底层类型的包的情况下实现-我们只是对它的行为感兴趣。

但这是每个人都应该在代码审查中抓住的简单东西。更根本的是这个代码的问题是，我不知道原始错误来自哪里。如果authenticate返回错误，则会AuthenticateRequest将错误返回给其调用者，他们可能会这样做，等等。在程序的顶部，程序的主体将把错误输出到屏幕或日志文件中，所有打印的内容是：No such file or directory。

没有文件和行的信息生成错误。调用堆栈没有导致错误的堆栈跟踪。此代码的作者将被迫进行长时间的平分其代码，以发现哪些代码路径触发了文件未找到错误。Donovan和Kernighan的Go编程语言建议你使用添加上下文到错误路径fmt.Errorf

但正如我们前面所看到的，这种模式与使用标记错误值或类型断言不兼容，因为将错误值转换为字符串，将其与另一个字符串合并，然后将其转换为具有fmt.Errorf中断等式的错误并销毁任何上下文在原来的错误。

最合适的错误处理策略

上面我们提到了三种错误处理策略，其中第三种策略耦合性最低。然而，第三种方式也存在一些问题：

  • 如何获得更详细错误信息，比如stack trace，帮助定位错误原因
  • 如何优雅的处理错误
    • 有些场景需要了解错误细节，比如网络调用，需要知道是否是瞬时的中断
    • 是否每层捕捉到错误的时候都需要处理

基本写法

我们在设计函数时，错误处理要遵循以下2个规则：

  1. 被调用函数如果有错误，需要传递给调用者，一定要返回
  2. 调用者收到返回的错误，一定不可忽视，忽视就是埋bug。如果调用者不处理，被调函数就需要设计成无错误返回。

	if y == 0 {
		return 0, errors.New("error in d, divided by 0")
	}
	return x / y, nil
}

test()函数是清爽了不少，也不用一直if然后处理err了，但是这种处理把错误转化为panic，导致test()后续的代码无法再继续执行，b(), c(), d()几个函数就没法执行了。

所以如果test()函数，遇到错误后就返回，就很适合这种优雅的方式。

当Golang的代码执行时，如果遇到defer的闭包调用，则压入堆栈。当函数返回时，会按照后进先出的顺序调用闭包。对于闭包的参数是值传递，而对于外部变量却是引用传递，所以闭包中的外部变量err的值就变成外部函数返回时最新的err值。

// 封装f：为传入的函数增加defer
func errorHandler(f func()) func() {
	return func() {
		defer func() {
			if r := recover(); r != nil {
				log.Println("got error: ", r)
			}
		}()

		f()
	}

func Cause(err error) error {
	type causer interface {
		Cause() error
	}

	for err != nil {
		cause, ok := err.(causer)
		if !ok {
			break
		}
		err = cause.Cause()
	}
	return err
}

使用for循环一直找到最根本（最底层）的那个error。

结论

简而言之，错误是你的软件包的公共API的一部分，与你公开API的任何其他部分一样谨慎。为了获得最大的灵活性，我建议你尝试将所有错误视为不透明。在不能这样做的情况下，为行为声明错误，而不是类型或值。最大限度地减少程序中的标记错误值的数量，并将错误转换为不透明的错误，只要它们一出现就用errors.Wrap将它们包装起来。最后，如果需要检查它，则使用errors.Cause来恢复基本错误。
